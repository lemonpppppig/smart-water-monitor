**讲稿要点**
- 硬件不是标准品：水源地、工业园区、跨界断面、农村水体四类场景对感知的**指标清单**、**采集频次**、**防护等级**、**能耗预算**需求截然不同
- 水源地重点看生物毒性与藻类爆发，工业园区重点盯重金属与特征污染物，跨界断面强调法定指标的行政可采信性，农村水体更关心低成本与低功耗
- 采样频次跨度从 **5 分钟** 到 **日均**，不同频次直接决定了主控定时器调度与能耗策略
- **技术事实**：没有一种「通用水站」，硬件采用"核心通用主控 + 场景化传感器模组"的可配置组合思路

**下一页过渡**：讲清楚场景差异后，接下来逐一看传感器怎么把这些物理量变成电信号。

## A-P2 · 水质传感器矩阵与测量原理

```
┌───────────────────────────────────────────────────────────────────────┐
│  A-P2                              水质传感器矩阵与测量原理            │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│   ┌────────────┬────────────────────────┬──────────┬────────────┐    │
│   │ 指标类别   │ 测量原理               │ 配置档位 │ 工程化要点 │    │
│   ├────────────┼────────────────────────┼──────────┼────────────┤    │
│   │ 常规五     │ pH：玻璃电极 / ΔV      │ 基础版   │ 参比年换   │    │
│   │ 参数       │ DO：荧光猝灭法         │ 必配     │ 膜片半年   │    │
│   │ (基础版)   │ 温度：PT1000 铂电阻    │          │ 稳定免校   │    │

│   ├────────────┼────────────────────────┼──────────┼────────────┤    │
│   │ 营养盐     │ 氨氮：离子选择/纳氏    │ 可选扩展 │ 试剂耗材   │    │
│   │ (可选)     │ 总氮：紫外消解+分光    │ 按需加装 │ 消解腔清洗 │    │
│   │            │ 总磷：钼酸铵分光光度   │          │ 高温高压   │    │
│   ├────────────┼────────────────────────┼──────────┼────────────┤    │
│   │ 有机物     │ CODMn：高锰酸盐滴定    │ 可选扩展 │ 批次校准   │    │
│   │ (可选)     │ CODCr：紫外 254nm      │ 按需加装 │ 替代手工   │    │
│   ├────────────┼────────────────────────┼──────────┼────────────┤    │
│   │ 藻类       │ 叶绿素 a：470nm 荧光   │ 可选扩展 │ 光窗防附着 │    │
│   │ (可选)     │ 蓝绿藻：620nm 蓝光     │ 按需加装 │ 背景扣除   │    │
│   └────────────┴────────────────────────┴──────────┴────────────┘    │
│                                                                       │
│   硬件按需选配 ⇅ 软件平台统一支持 11 项字段（ph/do/temp/conductivity/ │
│   turbidity/nh3_n/tn/tp/codmn/chla/...）即插即用                       │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**讲稿要点**
- **配置定位先讲清楚**：硬件端**只有常规五参数是基础版必配搭配**（pH、DO、温度、电导率、浊度），其余营养盐、有机物、藻类三类**全部是可选扩展**，按站点类型与预算按需加装
- **基础版五参数**用的都是物理法，稳定、耐用、免试剂，单站成本低、运维负担轻，适合一般监测断面与密集布点
- **可选 · 营养盐类**（氨氮、总氮、总磷）多用光学法 / 电化学法，需试剂瓶、需消解腔、需定期清洗，是耗材主要来源，一般部署在重点排口、富营养化敏感断面
- **可选 · 有机物**用 **CODCr 在线紫外 254nm 光谱法**替代传统手工高温滴定，避开强酸废液，通常加装在工业排污影响区域
- **可选 · 藻类监测**采用荧光法，蓝光激发 + 特定波长发射，对光窗清洁度要求高，重点部署在水华易发的湖库型水体
- **硬件可选 ⇅ 软件全支持**：即便现场站点只上五参数，后端数据模型与前端面板也按 **11 项字段**（ph/do/temp/conductivity/turbidity/nh3_n/tn/tp/codmn/chla/...）统一建模；缺失字段留空，不影响接入——**不同站点可以有不同传感器组合，平台侧零改动**
- **技术事实**：传感器选型不仅考虑精度，还要一并算上**校准周期、耗材消耗速率、清洗频次**——这三个指标直接决定站点运维成本模型，也是"基础/扩展"分档的根本依据

**下一页过渡**：传感器把物理量变成了电信号，接下来需要一个主控把它们统一采集、校验、转发。

## A-P3 · 主控选型与现场总线接口

```
┌───────────────────────────────────────────────────────────────────────┐
│  A-P3                                主控选型与现场总线接口            │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│   ┌──────────────────────────────────────────────────────────────┐    │
│   │            本项目主控：STM32F407VET6（LQFP-100）              │    │
│   │  Cortex-M4F @168MHz · FPU+DSP · 512KB Flash · 192KB SRAM     │    │
│   │  3×USART · 2×UART · 2×CAN · 3×12-bit ADC · DMA · FSMC · IWDG │    │
│   └──────────────────────────────────────────────────────────────┘    │
│                                                                       │
│     片内资源分配：                                                    │
│       USART1 ──→ MAX485 ──→ RS485 主机（Modbus-RTU）                 │
│       USART2 ──→ 4G/NB 模块（AT 指令）                                │
│       TIM2   ──→ Modbus 帧间隔 3.5 字符定时判定                       │
│       DMA    ──→ 收发缓冲搬运 · 释放 CPU                              │
│       CRC    ──→ 外设辅助（或软件实现 CRC16-MODBUS）                  │
│       IWDG   ──→ 独立看门狗 · 死锁兜底                                │
│                                                                       │
│                                                                       │
│     RS485 + Modbus-RTU 多传感器挂载（菊花链拓扑）                    │
│                                                                       │
│     [MCU]──[485]═══A/B═══[传感器 01]═══[传感器 02]═══…═══[120Ω]     │
│                      ↑                                                │
│                   偏置电阻保持空闲电平  /  终端电阻抑制反射           │
│                                                                       │
│     抗干扰：光电隔离 / 共模扼流 / 屏蔽线单点接地 / 4~20mA 兜底通道   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**讲稿要点**
- **主控**：STM32F407VET6，Cortex-M4F 内核 @168MHz、512KB Flash / 192KB SRAM、LQFP-100、自带单精度 FPU + DSP 指令
- **选型依据**：硬实时中断响应、丰富的 UART/USART 用于多路 RS485、DMA 与 CRC 外设支撑 Modbus 高频轮询、IWDG 做死锁兜底、供货稳定且 BOM 成本可控
- **相邻方案**（仅作对比理由，不做主线）：ESP32 偏低功耗 Wi-Fi 节点、RK3568 / 树莓派偏向边缘 AI 网关，本项目现场更看重**实时采集 + 抗干扰 + 确定性调度**，故选 F407
- **现场总线**：RS485 + Modbus-RTU 菊花链挂载多个传感器，主从寻址、终端 120Ω 匹配抑制信号反射、偏置电阻保持总线空闲电平、CRC16 帧校验
- **边缘预处理职责**：多路传感器轮询调度、异常值本地初判、丢包补报、按频次聚合、断网本地缓存
- **抗干扰设计**：光耦隔离信号、共模扼流圈、屏蔽线单点接地；对强电磁环境站点用 4~20mA 电流环作为兜底模拟通道

**下一页过渡**：数据在 MCU 里已就绪，下一步是走哪条路回到机房。

```
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**讲稿要点**
- **回传选型遵循场景**：城市近郊选 4G Cat.1（成本/带宽均衡），偏远山区选 NB-IoT（深覆盖 + 超低功耗），跨域无信号场景用北斗短报文兜底，园区内允许走以太网的直接有线
- **双通道冗余**：主用链路 + 备用链路自动切换，SIM 物联卡多运营商池化应对单运营商盲区
- **断点续传**：断网期间数据落本地 SD / Flash，网络恢复后回灌；每条数据带采集时间戳，避免上云顺序与时间顺序错位
- **OTA**：双分区固件 + 回滚机制，升级失败自动恢复上一版本，避免现场维护介入
- **时钟同步**：NTP（有网） / GPS 授时（授时卡）保证数据帧时间戳到秒级精度，为后续 AI 时序分析打基础
- **供电与防护**：市电 + 太阳能 + 锂电 UPS 三级供电；IP68 防水、防雷浪涌、宽温设计、近海场站额外考虑盐雾腐蚀

**下一页过渡**：回传链路有了，最后看数据到底按什么格式、带什么校验位往上走。

## A-P5 · 数据校验与 MQTT 上行协议

```
┌───────────────────────────────────────────────────────────────────────┐
│  A-P5                        数据校验与 MQTT 上行协议                  │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│   本项目不使用 HJ212 环保报报协议，现场采用：                         │
│     · 现场总线校验：Modbus-RTU + CRC16-MODBUS                         │
│     · 上行协议：MQTT 直推自定义 JSON                                  │
│                                                                       │
│   ┌─────────────────────── Modbus-RTU 帧结构 ────────────────────┐    │
│   │  ┌───────┬───────┬──────────────┬───────────────┐           │    │
│   │  │ 从机  │ 功能  │ 数据区（N B）│ CRC16(低位在前)│           │    │
│   │  │ 地址  │ 码    │              │                │           │    │
│   │  └───────┴───────┴──────────────┴───────────────┘           │    │
│   │                                                               │    │
│   │  CRC16-MODBUS 计算：                                          │    │
│   │     多项式 0xA001 · 初值 0xFFFF · 逐位异或与右移              │    │
│   │     传感器侧完成校验，MCU 收到后复算比对才入队                │    │
│   └───────────────────────────────────────────────────────────────┘    │
│                                                                       │
│   ┌─────────────────── 上行 MQTT JSON Schema ─────────────────┐       │
│   │  {                                                        │       │
│   │    "station_id":  "...",     // 站点编号                   │       │
│   │    "timestamp":   1714291200123,  // 毫秒                 │       │
│   │    "metrics": {              // 11 项水质字段              │       │
│   │       ph, do, temp, conductivity, turbidity,             │       │
│   │       nh3_n, tn, tp, codmn, chla, ...                    │       │
│   │    },                                                    │       │
│   │    "device_meta": { fw_ver, rssi, battery, ... }         │       │
│   │  }                                                        │       │
│   └───────────────────────────────────────────────────────────┘       │
│                                                                       │
│   MQTT 核心能力：Pub/Sub · QoS 0/1/2 · Retain · Will · Keep Alive     │
│   项目主题树：                                                        │
│     water/quality_data        ← 原始水质上报                          │
│     water/anomaly_detected    ← 边缘异常初筛                          │
│     water/alert_generated     ← 平台告警下发                          │
│   传输安全：MQTT over TLS + 客户端证书 / 用户名密码双因子             │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**讲稿要点**
- **首先纠正一个常见假设**：环保行业习惯的 HJ212 报文协议本项目**未使用**。现场只用 **Modbus-RTU + CRC16** 做链路层校验，**MQTT 直推自定义 JSON** 做应用层上行
- **Modbus-RTU 报文结构**：从机地址 + 功能码 + 数据区 + CRC16（低位在前）
- **CRC16-MODBUS 算法**：多项式 **0xA001**、初值 **0xFFFF**、逐位异或 + 右移；传感器出厂时固化，MCU 收到后复算校对，只有校验通过的帧才允许入应用队列
- **自定义 JSON 设计取舍**：相较 HJ212，在纯物联网场景下 JSON 更轻量、字段可演进、与后端领域模型直通；带 `station_id` + 毫秒级 `timestamp` 支撑幂等去重；`metrics` 承载 11 项水质字段；`device_meta` 携带固件版本、信号强度、电量等运维信息
- **MQTT 能力选择**：Pub/Sub 解耦设备与后端，QoS 0/1/2 按主题差异化、Retain 让新订阅者立即拿到最近一次快照、Will 遗嘱报文识别设备异常掉线
- **主题树分层**：`water/quality_data`（原始）、`water/anomaly_detected`（边缘初筛）、`water/alert_generated`（平台告警）三级分层，同时也是 EMQX 侧权限隔离边界
- **传输安全**：MQTT over TLS + 客户端证书或用户名密码做双因子鉴权

**衔接到 Part B**：「一条带 CRC16 校验的 Modbus 帧在现场拼装好、经 MCU 转成 JSON、由 MQTT 推上 EMQX 之后，它的下一站是后端。接下来看平台如何接纳、清洗、分流这条报文。」
