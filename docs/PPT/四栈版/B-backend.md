**讲稿要点**
- **架构口径**：项目已由原微服务（gateway + 5 个 service）整合为**模块化单体**——单一 FastAPI 进程内按业务域划分五个模块，统一部署、统一数据库会话池、统一生命周期
- **五大业务模块**：`app/data/`（MQTT 接入 + 时序落库） / `app/station/`（站点管理 + MQTT 绑定） / `app/alert/`（预警规则与触发） / `app/report/`（报告生成） / `app/ai/`（智能分析）
- **共享层** `backend/shared/`：数据库会话工厂、MQTT 客户端封装、跨模块领域模型、通用工具；所有模块复用同一套基础能力
- **选型理由**：当前团队规模与功能边界还不需要跨进程拆分；单体在**部署效率、跨模块事务、运维复杂度**上的优势对本项目更实际
- **演进设计**：每个模块保持独立的 `api/`、`services/`、`models/`、`database.py`，一旦某个模块需要独立扩容，按现有边界可直接切出为微服务
- **技术事实**：已弃用的 `backend/gateway/` 目录保留作多服务演进备选，但当前不纳入运行路径

**下一页过渡**：架构边界讲完了，下一步看这个进程的运行时是如何编排起来的。

## B-P2 · 单体运行时与数据接入层

```
┌───────────────────────────────────────────────────────────────────────┐
│  B-P2                     单体运行时与数据接入层                       │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│   FastAPI Lifespan 统一启动钩子                                       │
│   ┌─────────────────────────────────────────────────────────────┐     │
│   │  ① station.init_db()   → PostgreSQL 站点表初始化             │     │
│   │  ② alert.init_db()     → PostgreSQL 预警规则表初始化         │     │
│   │  ③ report.init_db()    → PostgreSQL 报告记录表初始化         │     │
│   │  ④ data.init_db()      → PostgreSQL 接入元数据 +             │     │
│   │                          TDengine 超级表初始化               │     │
│   │  ⑤ data.mqtt_bootstrap → 从 DB 加载连接配置 + 启动订阅线程   │     │
│   │  ⑥ ai.warmup           → 模型懒加载 / 热路径预热             │     │
│   └─────────────────────────────────────────────────────────────┘     │
│                                                                       │
│   路由层 · 六个 router 以 /api/v1 前缀统一挂载                        │
│     /api/v1/stations   /api/v1/data    /api/v1/mqtt                   │
│     /api/v1/alerts     /api/v1/ai      /api/v1/reports                │
│                                                                       │
│   多模块协作：                                                        │
│     事件驱动（MQTT 回调） + 同步调用（REST） + 后台任务（BG Tasks）   │
│                                                                       │
│   MQTT 订阅线程模型                                                   │
│                                                                       │
│       paho-mqtt                                                       │
│       ┌────────────┐    on_message 回调                               │
│       │ 独立 IO 线程│ ────────────────→ 回调队列（线程安全）          │
│       └────────────┘                         │                        │
│                                              ▼                        │
│                              ┌──────────────────────────┐             │
│                              │  asyncio 异步落库协程     │             │
│                              │  （幂等键：station_id +  │             │
│                              │    timestamp 复合主键）  │             │
│                              └──────────────────────────┘             │
│                                                                       │
│   上行报文解析：直接 json.loads → 映射到 11 项水质字段领域模型        │
│   （本项目不使用 HJ212，无报文字段映射层）                            │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**讲稿要点**
- **Lifespan 统一启动钩子**：FastAPI 的 `@asynccontextmanager` 把五个模块的 DB 初始化、MQTT 订阅、模型预热按顺序串起来，保证进程起来后所有子系统就绪
- **路由挂载**：`include_router(..., prefix="/api/v1")` 把六个 router 统一挂到版本前缀下；依赖注入用 `Depends` 把 DB/TDengine/Neo4j 会话递给处理函数，模块间不再各自管理连接
- **三种模块协作模式**：事件驱动（MQTT 收到消息触发）、同步调用（REST 跨模块 API）、后台任务（FastAPI BackgroundTasks 延后执行）
- **MQTT 订阅线程模型**：paho-mqtt 客户端运行在独立 IO 线程上，`on_message` 回调把消息塞进线程安全队列，asyncio 协程从队列取出做异步落库——避免阻塞事件循环
- **上行报文解析**：直接 `json.loads` 把 MQTT payload 反序列化，字段映射到系统领域模型的 11 项水质字段（ph / do / temp / conductivity / turbidity / nh3_n / tn / tp / codmn / chla / ...）——因为上行已是自定义 JSON，**没有 HJ212 字段映射层**
- **幂等与回压**：用 `station_id + timestamp` 作为复合主键去重；高峰期启用本地队列 + 批量写入回压缓冲，保护下游数据库

## B-P3 · DataProcessor 预处理管道

```
┌───────────────────────────────────────────────────────────────────────┐
│  B-P3                     DataProcessor 预处理管道                     │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  三段式管道（每条报文必经）                                           │
│                                                                       │
│   [MQTT JSON]                                                         │
│        │                                                              │
│        ▼                                                              │
│   ┌─────────────┐   ① 字段验证                                        │
│   │ Validator   │   · 必需字段（station_id / timestamp / metrics）    │
│   │             │   · 数值范围（物理合理性 · 国标阈值）               │
│   │             │   · 类型强转（str→float · int→datetime）            │
│   │             │   · 单位归一化（℃ · mg/L · NTU · μS/cm）           │
│   └──────┬──────┘                                                     │
│          │  valid / suspicious / invalid                              │
│          ▼                                                            │
│   ┌─────────────┐   ② 异常初筛                                        │
│   │ Isolation   │   · 基于随机树隔离路径深度                          │
│   │  Forest     │   · 异常点路径短 · 正常点路径长                     │
│   │             │   · 输出 anomaly_score ∈ [0, 1]                     │
│   └──────┬──────┘                                                     │
│          │                                                            │
│          ▼                                                            │
│   ┌─────────────┐   ③ 衍生指标                                        │
│   │  Derived    │   · DO 饱和度（Henry 定律 · 温盐压补偿）            │
│   │  Metrics    │   · TLI 富营养化指数（TN/TP/Chla 综合）             │
│   │             │   · WQI 水质综合标识指数（分级打分）                │
│   └──────┬──────┘                                                     │
│          │                                                            │
│          ▼                                                            │
│   [落库：TDengine + PostgreSQL + 事件分发]                            │
│                                                                       │
│   数据质量分级：                                                      │
│     valid       → 正常全路径                                          │
│     suspicious  → 仍落库 + 标记可疑 + 触发二次复核                    │
│     invalid     → 丢弃 + 告警 + 进死信                                │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**讲稿要点**
- **验证层**：先做结构校验（必需字段是否齐全）、再做值域校验（pH ∈ [0, 14]、DO ≥ 0 等物理合理性）、再做类型强转与单位归一化；这一层是脏数据的第一道闸门
- **IsolationForest 原理**：基于「异常点在随机分裂树中更容易被隔离」的假设，用多棵随机树测量每个样本被隔离时的平均路径深度——异常点路径短、正常点路径长，归一化后输出 **anomaly_score ∈ [0, 1]**
- **为什么这里做初筛**：后端入库前的无监督初筛开销小、免标注，能提前拦截明显异常；AI 引擎的 LSTM 重建误差判定留给**更复杂的时序模式**（D-P2）
- **衍生指标**：
  - **DO 饱和度**：根据 Henry 定律，用水温、盐度、气压补偿出当前实际氧饱和百分比
  - **TLI 富营养化指数**：以 TN、TP、Chla 等营养盐与藻类指标加权综合
  - **WQI 综合水质标识指数**：多指标按国标分级打分，输出一个 0~100 综合值给大屏呈现
- **数据质量分级**：valid / suspicious / invalid 三档——valid 正常走全路径；suspicious 仍落库但带标记，交由二次复核；invalid 进入死信队列 + 触发设备侧告警
- **技术事实**：这一层既是**数据清洗**也是**特征工程前置**——为 AI 引擎的时序分析和知识推理准备了可直接消费的语义字段

## B-P4 · 多模数据库选型与三库职责

```
┌───────────────────────────────────────────────────────────────────────┐
│  B-P4                     多模数据库选型与三库职责                     │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  为什么不用单一数据库：                                               │
│    时序高频写  ·  业务事务 + 空间 + 向量  ·  关系遍历                 │
│    三类存储模型根本不同 → 多模各司其职                                │
│                                                                       │
│  ┌────────────────┐  ┌────────────────────────────┐  ┌──────────────┐ │
│  │  TDengine 3.2  │  │ PostgreSQL 16              │  │  Neo4j 5.15  │ │
│  │  （时序）      │  │ + PostGIS + pgvector       │  │  （图谱）    │ │
│  ├────────────────┤  ├────────────────────────────┤  ├──────────────┤ │
│  │ 超级表 + 子表  │  │ 业务事务：stations /        │  │ 节点：       │ │
│  │ 站点 = 子表    │  │   alerts / reports         │  │  Station     │ │
│  │ TAGS：         │  │ 空间：                      │  │  PollutionSrc│ │
│  │   station_id   │  │   geography(Point, 4326)   │  │  PollutionTy │ │
│  │   type         │  │   + GiST 空间索引          │  │  Rule        │ │
│  │   basin        │  │ 向量：                      │  │  EmergPlan   │ │
│  │ 连续查询：     │  │   vector(16) + IVFFlat     │  │  Department  │ │
│  │   分钟/小时/日 │  │ Alembic 管 schema 迁移     │  │ 关系：       │ │
│  │ 保留策略：     │  │                             │  │  FLOWS_TO    │ │
│  │   原始  90 天  │  │                             │  │   (distance, │ │
│  │   分钟  1 年   │  │                             │  │   travel_t)  │ │
│  │   小时  永久   │  │                             │  │  HAS_RULE    │ │
│  │                │  │                             │  │  HAS_PLAN    │ │
│  │                │  │                             │  │  INVOLVES    │ │
│  └────────────────┘  └────────────────────────────┘  └──────────────┘ │
│       ▲                        ▲                           ▲          │
│       │ 水质时序写入           │ 业务 / 空间 / 案例向量    │ 流域拓扑 │
│                                                                       │
│  跨库一致性：领域事件解耦 · 避免分布式事务                            │
│  初始化脚本：                                                         │
│    infrastructure/docker/postgres/init/*.sql                          │
│    infrastructure/docker/neo4j/init/01~04.cypher                      │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**讲稿要点**
- **单库做不到**：水质高频时序（每分钟 × 千站）、业务事务（站点/规则/报告）、空间查询（附近站点）、向量检索（相似案例）、图遍历（流域上下游）——存储模型差异太大，硬塞单一库会在任一维度都做不深
- **TDengine 3.2**：以「超级表 + 子表」组织，每个站点作为超级表下的一个子表，TAGS 列（station_id / type / basin）支撑跨站聚合；连续查询做**分钟 / 小时 / 日级**降采样；**保留策略**：原始数据 90 天 / 分钟级 1 年 / 小时级永久，兼顾成本与历史可溯
- **PostgreSQL 16 + PostGIS + pgvector**：业务事务承载 station / alert / report 三张主表；`geography(Point, 4326)` + **GiST 空间索引**做附近站点、落域归属；`vector(16)` + **IVFFlat 索引**做历史案例向量检索；Alembic 管理 schema 迁移
- **Neo4j 5.15**：流域拓扑以图建模——节点有 Station、PollutionSource、PollutionType、Rule、EmergencyPlan、Department；关系中 `FLOWS_TO` 携带 distance、travel_time 属性用于溯源计算，`HAS_RULE / HAS_PLAN / INVOLVES` 把污染类型绑定到规则、预案和责任部门
- **跨库一致性**：不引入两阶段提交等分布式事务，而是用**领域事件解耦**（MQTT 消息 + 后台任务）保证最终一致；业务方关心的强一致都落在单库
- **技术事实**：初始化依赖 `infrastructure/docker/postgres/init/*.sql` 与 `infrastructure/docker/neo4j/init/01~04.cypher` 在容器首次启动时注入 schema 与种子数据

**下一页过渡**：数据库讲完，最后看单体怎么对外、前端怎么进来、MQTT 怎么走。

**讲稿要点**
- **唯一对外入口**：模块化单体 `backend/app/main.py` 直接监听 8000 端口；**没有独立网关进程**前置；原 `backend/gateway/` 代码已合并入单体后弃用，仅作多服务演进备选保留
- **路由真实结构**：六个 router 通过 `include_router(..., prefix="/api/v1")` 统一挂载——`/api/v1/stations`、`/api/v1/data`、`/api/v1/mqtt`、`/api/v1/alerts`、`/api/v1/ai`、`/api/v1/reports`；另暴露 `/health` 健康检查与 `/docs` OpenAPI
- **应用生命周期**：FastAPI `lifespan` 上下文按顺序初始化 station / alert / report / data 四个 PostgreSQL 库，并在 data 阶段加载 MQTT 连接配置启动后台订阅线程
- **跨域策略（真实）**：仅启用 `CORSMiddleware`，`allow_origins=["*"]` + `allow_credentials/methods/headers=*`，**未配置白名单与预检缓存**——演讲口径是「演示阶段全开，生产再收敛」
- **开发环境链路**：Vite `server.proxy` 将 `/api`、`/health` 转发到 `http://127.0.0.1:8000`——强制 127.0.0.1 规避 Windows 下 `localhost` 解析为 IPv6 `::1` 导致的 socket hang up
- **生产环境链路**：Nginx 反向代理 `/api/` → backend，并在 Nginx 层做限流 `limit_req zone=api burst=20 nodelay`、统一超时与转发头；通用 `/ws/` 路径透传给后端的实时推送通道
- **MQTT 真实接入路径**（不是前端直连 WebSocket）：
  - 前端通过 REST `/api/v1/mqtt/*` 管理「外部 MQTT 数据源」连接配置（`frontend/src/services/api.ts` 的 `mqttApi` 仅 HTTP 调用）
  - 后端 paho-mqtt 客户端走 TCP **1883** 订阅 EMQX，数据落库后再以 HTTP/WebSocket 形式下发给前端
  - Nginx 未对外开放 EMQX `8083` 的 MQTT-over-WebSocket 端口
